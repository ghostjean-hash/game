# 단어 데이터 작성 규칙 (뜻·품사·예문·관련형) - 확정본 v1

단어 카드 내용(뜻·품사·예문·관련형)의 출처와 작성 규칙을 고정한다. 단어 선정 근거·전체 계획은 `vocab-master-plan.md`, 원천은 `sources/`. 확정 근거: 사용자 지시 + ChatGPT 제작 지시서(2026-07-24) 반영.

적용 대상: 우선 초등(공식 2022 `*` 대표형 800). 이후 중·고 동일.

## 0. 핵심 개념: 제작 배치 ≠ 학습 세트 (중요)

- **AUTHORING BATCH**(제작 배치): 알파벳 순 200개씩(01~04). 콘텐츠 제작·검수·재현의 작업 단위일 뿐이다.
- **LEARNING SET**(학습 세트): 사용자가 실제로 학습하는 SET 01~04. 순서는 알파벳이 아니라, 800개 전체 검수 후 빈도·활용도·난이도로 **별도 설계**한다.
- 알파벳 순은 제작·검증·중복/누락 검사·진행관리에만 쓴다. 학습 세트 순서로 확정하지 않는다(기억 간섭·글자/품사 편중 방지).
- 현재 단계 = **AUTHORING BATCH 01(알파벳 1~200)** 제작. 이건 LEARNING SET 01이 아니며 앱에 적용하지 않는다.

## 1. 카드 단위

1.1. **대표형 1개 = 학습 카드 1개.** 공식 2022 `*` 대표형 800개.
1.2. 파생형·관련형·이형은 별도 카드가 아니라 그 카드의 **보조 정보**(진도 중복 집계 안 함).
1.3. 보조 정보는 정답 공개(ANSWER) 카드에서 뜻 아래 작게 표시.

## 2. 출처 계층 분리 (메타데이터에 반드시 구분 기록)

- `officialSource`: 교육부 공식 영어 표제어 + 권장 학교급 별표(2022 별책14). **단어·별표의 유일 근거.**
- `lexicalReference`: 품사·일반 의미를 대조한 자유 라이선스 공개 자료(위키낱말사전, Wiktionary 등 복수 교차). 특정 사전 하나를 절대 기준으로 삼지 않는다. 의미 사실 확인·오선정 방지 용도.
- `authoredContent`: 앱용 대표 뜻·관련형·예문·예문 번역(자체 작성).

저작권: 상용/공개 사전 뜻풀이 문장을 그대로 복사·대량 인용 금지. 사실 확인 근거로만 쓰고, 최종 뜻은 짧고 자연스러운 학습용 표현으로 재작성. 예문·번역은 직접 작성(기계번역 결과 그대로 금지). 공개 사전 대조했다는 이유만으로 정확하다고 자동 처리하지 않는다(검수 필수).

## 3. 대표 뜻(meaningKr) 규칙

3.1. 초등 단계에서 필요한 **대표 뜻 1개 우선, 최대 2개**. 품사 관계가 분명해야 함.
3.2. 중학 이후 주로 배우는 의미·드문/전문 의미 제외. 사전식 장문 설명 금지.
3.3. 나쁜 예: light → "빛, 광선, 조명, 불빛, 밝음, 가벼운, …"(무분별 나열). 좋은 예: light → "빛 / 가벼운"(둘 다 초등에 중요할 때만).

## 4. 품사(partOfSpeech) 규칙

4.1. 실제 카드 대표 뜻과 **일치**하는 품사만. 여러 품사 가능해도 초등에 안 중요하면 다 넣지 않음.
4.2. 내부 표기값(프로젝트 통일): `noun/verb/adjective/adverb/pronoun/preposition/conjunction/determiner/interjection/auxiliary/number`. (화면에 한국어 품사명을 쓸지는 앱 적용 단계 UI 정책.)
4.3. 원천엔 품사 없음 → 공개 자료 대조로 확정, 지어내지 않음.

## 5. 관련형(relatedForms)·불규칙형(irregularForms)

5.1. `relatedForms`: 공식 괄호 파생형 중 초등 독해 빈출 + 대표형만으론 알아보기 어려운 것. **기본 0개, 최대 2개.** 가능한 파생어 전부 나열 금지, 중·고 수준 파생어 초등 카드에 넣지 않음.
5.2. `irregularForms`: 초등에 중요한 불규칙 활용(go → went, gone)은 별도 필드로.
5.3. 관련형·불규칙형은 진도 카드 수에 중복 포함하지 않음.

## 6. 예문(example/exampleKr) 규칙

6.1. 예문은 자작 불가피, 사람 검수가 유일한 관문. 목표 단어(또는 적절한 활용형) 포함.
6.2. 길이: 영어 **4~10단어**(최대 12). 초등 학습자가 이해 가능하고 뜻을 추론할 문맥. 지나치게 유아적이지 않게.
6.3. 어려운 새 단어 과다 금지, 고유명사·브랜드 의존 최소화, 편견 최소화, 번역투 금지. 억지로 끼워 넣은 문장 금지.
6.4. `exampleKr`은 자연스러운 한국어.

## 7. ID·데이터 버전·저장 키 (ID를 세트 위치와 분리)

7.1. **안정적 단어 ID**를 세트 위치와 분리한다(순서 조정 시 진도 유지). 형식: `ev-moe2022-e-0001` … `ev-moe2022-e-0800`(제작 시 알파벳 sourceOrder 기준 채번, 학습 순서와 무관하게 불변).
7.2. `setId`·`learningOrder`는 별도 필드. 학습 세트 확정 시 채운다(그전엔 null). `sourceOrder`(알파벳 제작순)와 `learningOrder`(최종 학습순)를 같은 값으로 간주하지 않는다.
7.3. 데이터 버전: `moe2022-elementary-v1`. 저장 키 네임스페이스 분리: `deck:moe2022:elementary:v1:*`. 기존 playland 진도와 혼합하지 않음.
7.4. **콘텐츠 수정과 진도 초기화 분리**: 오탈자·예문·뜻 표현 경미 수정·품사 오류·관련형 보완은 버전 안 올림(단어 ID 유지 시 진도 유지). 버전 상향은 표제어 구성 변경·대규모 증감·ID 체계 변경·구조 변경·세트 대개편 시에만.

## 8. 카드 스키마 (제작 배치용)

```
{
  "id": "ev-moe2022-e-0001",
  "sourceTier": "elementary",
  "sourceMarker": "*",
  "word": "example",
  "partOfSpeech": ["noun"],
  "meaningKr": ["예시"],
  "relatedForms": [],
  "irregularForms": [],
  "example": "This is a good example.",
  "exampleKr": "이것은 좋은 예시이다.",
  "sourceOrder": 1,
  "setId": null,
  "learningOrder": null
}
```
(앱 현행 스키마 `pos`(한국어)·set 구조와는 앱 적용 단계에서 매핑한다. 제작 배치 데이터는 이 리치 스키마를 쓴다.)

## 9. 2회 내용 검수 (같은 검증 2번 반복은 검수 아님)

배치별로 두 관문을 모두 통과해야 한다.
9.1. **1차(사실·형식)**: 공식 표제어 일치 / 별표 확인 / 품사 정확 / 대표 뜻 적합·초등 난이도 적합 / 관련형 필요성 / 예문 문법·뜻 일치·길이 / 번역 자연스러움 / `validate-data.mjs` 통과.
9.2. **2차(품질, 1차 수정 후 전체 재검토)**: 단어 간 뜻 표현 일관성 / 중복 뜻·예문 패턴 반복 / 인물·주제 편중 / 어려운 문법·어휘 혼입 / 번역투 / 카드 간 품질 편차 / 화면 학습 적합성(browser-shot 표본). 검수 결과·수정 이력 기록.

## 10. 금지

- 200개(또는 800개)를 한 번에 대량 생성 후 자동검사만 하고 완료 처리.
- 여러 에이전트가 서로 다른 기준으로 병렬 작성, 표본만 보고 전체 승인.
- 오류 발견 후 같은 규칙으로 다음 배치 강행. 공개 사전 대조 생략. 기계번역 예문 그대로.
- **앱 적용 금지(이 단계)**: 785 삭제·800 교체·진도 초기화·메뉴/세트 변경·배치를 LEARNING SET으로 등록·실사용 배포·다음 배치 자동 착수. 콘텐츠 제작과 앱 적용은 별도 단계.

## 11. 진행 게이트

AUTHORING BATCH 01(알파벳 1~200) 제작 + 2회 검수 완료 → 결과 보고 → **사용자 승인 후** BATCH 02 진행. 800 전체 검수 완료 후 LEARNING SET 순서·앱 적용을 별도 단계로 설계.
