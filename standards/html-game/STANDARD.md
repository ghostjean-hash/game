## 1. 프로젝트 개요

GitHub Pages로 호스팅하는 정적 HTML 게임을 만든다. 빌드 시스템 없이 브라우저가 ES Modules를 직접 실행하는 방식.

작업 방식은 문서 우선. 모든 요구사항은 문서를 먼저 수정하고, 그에 따라 코드를 작성한다. 문서가 진실의 원천(SSOT)이다.

## 3. 각 문서의 역할 (절대 혼동 금지)

3.1. `CLAUDE.md` - 매 세션 진입점. 인덱스와 절대 규칙. 1페이지 이내 유지.
3.2. `docs/01_spec.md` - 게임 사양. 규칙, 조작, 화면 흐름, 메커닉 정의.
3.3. `docs/02_data.md` - 게임 데이터. 모든 수치와 게임용 색상의 정의. 매직 넘버는 코드에 0개여야 함.
3.4. `docs/03_architecture.md` - 폴더/모듈 구조, 의존성 방향, 데이터 흐름.
3.5. `docs/04_conventions.md` - 네이밍, 주석, 테스트, 디자인 토큰 사용 규칙.
3.6. `README.md` - 사람용. 플레이 링크, 조작법, 로컬 실행법만. 짧게.

## 4. 절대 규칙

### 4.1. 문서 우선 워크플로우

- 사용자 요구사항 → 영향받는 문서 식별 → 문서 수정 → 코드 수정 → 테스트 수정 순서.
- 문서와 코드가 충돌하면 문서가 진실. 코드를 문서에 맞춤.
- 같은 정보를 두 문서에 적지 않는다. 한 곳만 SSOT, 나머지는 "→ 해당 문서 참조".

### 4.2. 기술 환경

- 빌드 시스템 없음. ES Modules 직접 사용 (`<script type="module">`).
- import는 상대경로 + `.js` 확장자 명시.
- TypeScript, 번들러, 트랜스파일러, npm 설치 금지.
- 외부 라이브러리는 가능한 vanilla로. 불가피하면 esm.sh / jsdelivr CDN의 ESM 빌드만.
- 폴리필 사용 안 함. 최신 Chrome / Safari / Firefox 기준.

### 4.3. 코드 구조

- `core/`는 DOM, Canvas, window, document 일체 import 금지. 순수 함수 위주.
- 게임 로직과 렌더링은 절대 한 모듈에 두지 않는다.
- 매직 넘버 금지. 모든 수치는 정의된 상수에서 가져옴.

### 4.4. 색상과 스타일

- 게임 데이터(블록 색, 이펙트 색 등 게임 플레이의 일부): `src/data/colors.js`의 상수만 사용.
- 디자인 토큰(메뉴 / HUD 색, 간격, 폰트 등 UI 요소): `styles/tokens.css`의 CSS 변수만 사용.
- 인라인 매직 값 금지. 필요하면 정의 위치에 먼저 추가 후 사용.

### 4.5. 데이터 저장

- 영속 데이터는 localStorage. 키 prefix는 게임명 (예: `tetris_`).
- 저장 스키마는 `docs/02_data.md`에 정의.

### 4.6. 테스트

- 브라우저 기반 테스트. `tests/test.html`을 열면 모든 테스트 자동 실행.
- `core/` 모듈은 테스트 필수. 특히 게임 핵심 로직 변경 시 반드시 테스트 함께 수정.

## 5. CLAUDE.md 템플릿

게임명과 한 줄 설명만 빈칸으로 두고, 나머지는 그대로.

```markdown
# [게임명]

## 1. 게임 한 줄
[한 문장 설명]

## 2. 작업 워크플로우 (반드시 지킬 것)
1. 사용자 요구사항을 받으면, 먼저 어떤 문서를 수정해야 하는지 식별
2. 해당 문서를 수정 (단 하나의 진실의 원천)
3. 변경된 문서 기반으로 코드 수정
4. 테스트 코드도 함께 수정

## 3. 문서 인덱스
- docs/01_spec.md - 게임 규칙, 조작, 화면 흐름
- docs/02_data.md - 게임 데이터, 수치, 색상
- docs/03_architecture.md - 폴더/모듈 구조, 의존성 규칙
- docs/04_conventions.md - 네이밍, 주석, 테스트, 토큰 규칙

## 4. 절대 규칙
- 게임 로직과 렌더링은 절대 한 모듈에 두지 않는다
- 매직 넘버 금지. 모든 수치는 docs/02_data.md 정의 상수에서
- core/는 DOM/Canvas/window/document 일체 import 금지
- 핵심 로직 변경 시 반드시 테스트 코드 업데이트
- docs와 코드가 충돌하면 docs가 진실. 코드를 docs에 맞춰 수정

## 5. 기술 환경
- ES Modules 직접 사용. 빌드/번들러/TypeScript 금지
- import는 상대경로 + .js 확장자 명시
- 외부 라이브러리 최소화. 필요시 CDN ESM 빌드만
- 영속 데이터는 localStorage, 키 prefix는 [게임명_]

## 6. 색상과 스타일
- 게임 데이터(블록 색 등): src/data/colors.js 상수만 사용
- 디자인 토큰(UI 색/간격/폰트): styles/tokens.css 변수만 사용
- 인라인 매직 값 금지

## 7. 실행
- 로컬: `python -m http.server` 후 localhost:8000
- 테스트: tests/test.html을 브라우저에서 열기
- 배포: main 브랜치 push → GitHub Pages 자동 배포

## 8. 변경 이력
- YYYY-MM-DD: html-game v<x.y> 적용 (초기 셋업)
```

## 6. 초기 작업 순서

각 단계 완료 후 보고.

6.1. 위 폴더 구조와 빈 파일들 생성 (`.nojekyll` 포함).
6.2. CLAUDE.md를 위 템플릿으로 작성 (게임명/설명은 비워두기).
6.3. 빈 index.html과 main.js로 "검은 화면이라도 뜨는지" 확인 가능한 최소 셋업 작성.
6.4. tests/test.html과 runner.js를 외부 라이브러리 없이 동작하는 최소 테스트 러너로 작성.
6.5. styles/tokens.css에 디자인 토큰 기본 골격 작성 (색 / 간격 / 폰트 변수, 빈 값이라도).
6.6. README.md에 로컬 실행법, GitHub Pages 배포법, 테스트 실행법 작성.
6.7. docs/ 4개 파일은 H1 제목과 섹션 골격만 작성 (내용은 게임 결정 후 채움).

## 7. 문서 비대화 대응 규칙

작업 진행 중 한 문서가 커지면 다음 순서로 대응한다. 임의로 분할하지 말 것.

7.1. 먼저 압축 시도: 중복 제거, 결정 외 맥락 제거, 산문을 표로.
7.2. 데이터를 코드로 분리: 수치 표가 길면 `src/data/*.js`로 빼고 문서엔 스키마와 경로만.
7.3. 그래도 한 문서가 500줄을 넘고 변경 빈도가 명확히 갈리면 해당 문서만 폴더로 분할.

예: `docs/02_data.md` → `docs/02_data/` 폴더 + `_index.md` + 분할 파일들.
- 다른 문서는 단일 파일 유지.
- 분할 시 CLAUDE.md 인덱스 동기화 필수.

## 8. 게임 컨셉 결정 단계

위 셋업 완료 후, 다음 항목을 사용자에게 질문해서 `docs/01_spec.md`와 `docs/02_data.md`를 채운다. 사용자 답변 전에 임의 결정 금지.

8.1. 게임 장르와 한 줄 컨셉.
8.2. 핵심 메커닉 (코어 루프).
8.3. 조작 방식 (키보드 / 마우스 / 터치).
8.4. 성공 이벤트 / 실패 이벤트 / 종료 조건 (해당 없음 가능).
- 통상 게임 예: 클리어 = 성공, 게임오버 = 실패, 종료 = 클리어 시점.
- 추천기 / 시뮬레이션 예: 적중 = 성공 이벤트, 실패 없음, 종료 없음.
- 컬렉션 게임 예: 도감 채우기 = 성공, 실패 없음, 종료 없음.
8.5. 진행 시스템 (스테이지 / 스코어 / 시간 등).
8.6. 모바일 지원 여부.
