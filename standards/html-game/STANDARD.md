## 1. 프로젝트 개요

GitHub Pages로 호스팅하는 정적 HTML 게임을 만든다. 빌드 시스템 없이 브라우저가 ES Modules를 직접 실행하는 방식.

작업 방식은 문서 우선. 모든 요구사항은 문서를 먼저 수정하고, 그에 따라 코드를 작성한다. 문서가 진실의 원천(SSOT)이다.

## 4. 절대 규칙

### 4.1. 문서 우선 워크플로우

- 사용자 요구사항 → 영향받는 문서 식별 → 문서 수정 → 코드 수정 → 테스트 수정 순서.
- 문서와 코드가 충돌하면 문서가 진실. 코드를 문서에 맞춤.
- 같은 정보를 두 문서에 적지 않는다. 한 곳만 SSOT, 나머지는 "→ 해당 문서 참조".

### 4.2. 기술 환경

- 빌드 시스템 없음. ES Modules 직접 사용 (`<script type="module">`).
- import는 상대경로 + `.js` 확장자 명시.
- TypeScript, 번들러, 트랜스파일러, npm 설치 금지.
- 외부 라이브러리는 가능한 vanilla로. 불가피하면 esm.sh / jsdelivr CDN의 ESM 빌드만.
- 폴리필 사용 안 함. 최신 Chrome / Safari / Firefox 기준.

### 4.3. 코드 구조

- `core/`는 DOM, Canvas, window, document 일체 import 금지. 순수 함수 위주.
- 게임 로직과 렌더링은 절대 한 모듈에 두지 않는다.
- 매직 넘버 금지. 모든 수치는 정의된 상수에서 가져옴.

### 4.4. 색상과 스타일

- 게임 데이터(블록 색, 이펙트 색 등 게임 플레이의 일부): `src/data/colors.js`의 상수만 사용.
- 디자인 토큰(메뉴 / HUD 색, 간격, 폰트 등 UI 요소): `styles/tokens.css`의 CSS 변수만 사용.
- 인라인 매직 값 금지. 필요하면 정의 위치에 먼저 추가 후 사용.

### 4.5. 데이터 저장

- 영속 데이터는 localStorage. 키 prefix는 게임명 (예: `tetris_`).
- 저장 스키마는 `docs/02_data.md`에 정의.

### 4.6. 테스트

- 브라우저 기반 테스트. `tests/test.html`을 열면 모든 테스트 자동 실행.
- `core/` 모듈은 테스트 필수. 특히 게임 핵심 로직 변경 시 반드시 테스트 함께 수정.

### 4.7. 모바일/태블릿 브라우저 플레이 규칙

핸드폰·태블릿 브라우저에서 플레이하는 게임은 아래를 지킨다(노노그램에서 검증, v0.3 신설 2026-07-05). 규칙 1-4의 무테마 골격(색·폰트 없이 뷰포트 고정·safe-area·터치·리셋만)은 공용 자산 `shared/mobile-shell.css`가 제공한다(v0.3.1 실제 분리). 다크 테마 게임은 `shared/base.css`를 링크하면 되고(base.css가 mobile-shell.css를 `@import`), 자체 디자인 게임(파스텔 톤 등)은 `shared/mobile-shell.css`를 직접 링크한 뒤 자기 색·폰트만 얹는다. 규칙 값을 복사하지 않는다.

1. **viewport meta**: `width=device-width, initial-scale=1, viewport-fit=cover`. 게임판을 드래그로 조작하는 게임은 `maximum-scale=1, user-scalable=no`로 확대를 잠근다. `viewport-fit=cover`가 있어야 safe-area가 동작한다.
2. **페이지 고정**: `html { overflow:hidden }`, `body { height:100dvh; overscroll-behavior:none; touch-action:manipulation }`. 페이지 자체가 세로 스크롤·고무줄 바운스·더블탭 확대되지 않게 뷰포트에 고정한다. `100vh`가 아니라 `100dvh`(모바일 주소창 변화 대응).
3. **safe-area**: `body`에 `env(safe-area-inset-*)` 패딩. 노치·홈바에 UI가 가리지 않게 안전 영역 안에만 둔다.
4. **드래그 스크롤 차단**: 게임판(드래그 조작 영역)은 `touch-action:none`, 게임판을 감싼 컨테이너는 `overflow:hidden`. 드래그해도 판이 좌우·상하로 밀려 스크롤되지 않는다. 목록이 긴 화면(맵·결과 등)만 그 화면의 내부 컨테이너에서 스크롤한다(`overflow-y:auto`) - 페이지가 아니라 그 안에서.
5. **방향별 레이아웃**: 세로에서는 UI를 위·아래로, 가로에서는 좌·우로 분산 배치하고 게임판을 화면 중앙에 둔다(`@media (orientation:landscape)`로 재배치). 가로에서는 세로 폰 기준 폭 제한(`max-width`)을 풀어 넓은 화면을 활용하고, 좌·우 UI 열을 같은 폭으로 맞춰 게임판이 정중앙에 오게 한다.
6. **전체화면**: 지원 기기에서 Fullscreen API 토글 버튼을 제공하고, 미지원(iOS Safari 등)이면 버튼을 숨긴다.
7. **보드 크기 산정 - 창 비율 무관 전체 fit**: 셀/보드 크기는 뷰포트 폭·높이를 둘 다 재서 남는 공간에 맞춘다(폭만 기준으로 하면 가로 방향에서 높이를 넘겨 스크롤이 생긴다). 가용 폭에는 보드 옆에 놓이는 UI 열(버튼·상태 표시) 폭까지 포함해 뺀다 - 어느 한 축을 "병목 아님"으로 가정하면(예: 가로에서 폭 무시) 데스크톱 창 리사이즈처럼 임의 비율 창에서 게임이 화면 밖으로 잘린다. 창/뷰포트 비율이 어떻든 게임 UI 전체가 스크롤·잘림 없이 한 화면에 들어와야 한다(노노그램 가로 창 좌우 잘림 사례, v0.3.2). 방향 전환·리사이즈·전체화면 전환 시 재계산한다.

## 6. 초기 작업 순서

각 단계 완료 후 보고.

6.1. 위 폴더 구조와 빈 파일들 생성 (`.nojekyll` 포함).

6.6. README.md에 로컬 실행법, GitHub Pages 배포법, 테스트 실행법 작성.
6.7. docs/ 4개 파일은 H1 제목과 섹션 골격만 작성 (내용은 게임 결정 후 채움).

## 8. 게임 컨셉 결정 단계

위 셋업 완료 후, 다음 항목을 사용자에게 질문해서 `docs/01_spec.md`와 `docs/02_data.md`를 채운다. 사용자 답변 전에 임의 결정 금지.

8.1. 게임 장르와 한 줄 컨셉.
8.2. 핵심 메커닉 (코어 루프).
8.3. 조작 방식 (키보드 / 마우스 / 터치).
8.4. 성공 이벤트 / 실패 이벤트 / 종료 조건 (해당 없음 가능).
- 통상 게임 예: 클리어 = 성공, 게임오버 = 실패, 종료 = 클리어 시점.
- 추천기 / 시뮬레이션 예: 적중 = 성공 이벤트, 실패 없음, 종료 없음.
- 컬렉션 게임 예: 도감 채우기 = 성공, 실패 없음, 종료 없음.
8.5. 진행 시스템 (스테이지 / 스코어 / 시간 등).
8.6. 모바일 지원 여부.
