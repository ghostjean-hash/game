# 01. 게임 사양

## 1. 게임 한 줄

통계 데이터 + 캐릭터 시드 기반 한국 6/45 로또 번호 추천 RPG. 당첨 확률 향상이 아니라 선택 행위의 서사화.

## 2. 코어 루프

1. 캐릭터 선택 (또는 첫 진입 시 생성).
2. 회차 / 운세 확인.
3. 추천 카드 생성 (본번호 6 + 보너스 1).
4. 구매 인증 (선택, 실제 구매 후 사용자가 토글).
5. 다음 회차 결과 발표 후 자동 매칭.
6. 캐릭터 전적 / Luck 갱신.

## 3. 조작

3.1. 터치 우선 (모바일 PWA).
3.2. 키보드 보조: Tab 이동, Enter 확정, Esc 모달 닫기, 방향키 캐릭터 카드 전환.
3.3. 마우스는 데스크톱에서 터치와 동일 동작.

4.1. **첫 진입**: 면책 모달 → 캐릭터 생성 폼(직접) → 추첨 탭.
4.2. **이후 진입**: 면책 스킵 → 추첨 탭 직행.
4.3. **추가 캐릭터**: 슬롯의 `+` → 모달 안 폼 → 추가 후 자동 활성화.
4.4. **캐릭터 전환**: 슬롯 클릭 → 활성 변경 → 즉시 추천/카드 갱신.
4.5. **활성 캐릭터 삭제**: 슬롯의 `×` → 확인 → 첫 캐릭터로 활성. 마지막 1명일 때 비활성.
4.6. **탭 전환**: 하단 탭 클릭 → `state.currentTab` 갱신 → 본문만 다시 렌더 (탭 / 헤더는 유지).
4.7. **전략 변경**: 추첨 탭의 카테고리별 그룹 전략 탭에서 직접 클릭 → 즉시 활성 변경 + 캐릭터 `lastUsedStrategy` 저장 + 재렌더. 시트 모달은 폐기.

4.7.1. **시각 (S9 + S10 + S11, 2026-05-02)**: 11개 전략 버튼을 카테고리별로 3줄에 묶어 노출. 줄 순서 = 운세(3) → 랜덤(3) → 통계(5). 캐릭터 정체성/콘텐츠 우선, 객관 통계는 후순위(S11). 줄 앞에 카테고리 라벨 chip(카테고리 색). 한 줄 폭 초과 시 wrap (가로 스크롤 미사용). 활성 전략 설명은 그룹 아래에 표시.
4.7.2. **카테고리 라벨 chip**: 줄 시작에 작은 chip(`.strategy-row-label`). 카테고리 색 cascade(`is-stats` / `is-mapping` / `is-random`). 클릭 불가, aria-hidden. 라벨 텍스트는 "통계" / "운세" / "랜덤".
4.7.3. **wrap 정책**: 카테고리 줄 컨테이너는 `display: flex; flex-wrap: wrap`. 한 줄당 1~5개라 데스크톱 / 표준 모바일 모두 한 줄 fit, 좁은 화면에서 자연 줄 바꿈.
4.7.4. **폐기 항목 (S9 이전 가로 스크롤 정책)**: 단일 줄 가로 스크롤이 카테고리 그룹화로 대체되어 다음이 모두 불필요해 폐기: 컨테이너 양옆 mask fade(`is-start` / `is-end`), `scrollLeft` 모듈 변수 보존, 활성 탭 잘림 보정, PC 휠 → 가로 변환. 스크롤바 숨김(`scrollbar-width: none`)은 wrap 컨테이너에 가로 스크롤 미발생이라 자동 무관.
4.7.5. **활성 / 비활성 폭 변동 방지**: 활성 / 비활성 탭 모두 동일 `font-weight: 600`로 클릭 시 폭 변동 0 (S9 이전 정책 유지).
4.8. **다구좌 활성화**: 휠링 탭 진입 시 OFF면 안내 + 켜기 버튼 → 윤리 모달 → ON 후 휠링 본문. 또는 설정 탭에서도 토글 가능.

## 5. 메커닉

### 5.1. 캐릭터

- 캐릭터 = **정체성** (이름 + 시드 + 띠 + 별자리 + Luck + 이력). 생성 시 1회 결정.
- 시드: 사용자 입력(생년월일 / 이름 / 행운의 단어) + 자동 계산된 별자리(02_data.md 2.6) 합쳐 결정론적 해시.
- Luck: 가중치 분산도 조절기 (0 = 균등, 100 = 시드 6개로 극도 쏠림). 당첨 확률은 동일.
- 운세 사이클: 회차별 컨디션 등급 (대길 / 길 / 평 / 흉). 캐릭터 12간지(생년 기반) + 회차 일진의 관계로 분포 결정 (5.1.1) + 캐릭터 일주(생년월일) + 회차 일주의 천간 오행 관계로 보정 (5.1.4 참조).

### 5.1.3.0. 추천 알고리즘 architecture (S43, 2026-05-08 재구축)

**핵심 결정**: 사용자 통찰 "알고리즘 근본 자체 잘못. 부분 fix 의미 없다" 반영. 옛 architecture 폐기 + 단일 추첨 architecture 도입.

#### 옛 architecture 결함 (S0~S42 누적 결함)

- 다중 strategy 분배(`distributeCounts`)로 6번호를 N개 strategy에 2개씩 균등 분할 → 각 풀 별도 추첨 → 정렬 시 끝자리 패턴 충돌(2/3/4, 21/22/24 인접 클러스터링).
- 풀 컷팅(`STATS_POOL_SIZE`)으로 풀 좁아 결정론 채택. 풀 안 1-9 비중이 그대로 노출.
- Luck 25배 보너스(`WEIGHT_MAX_BIAS = 50`)로 시드 6번호 채택 84%. 캐릭터 시드에 작은 번호 포함 시 무한 반복.
- balancer 합 필터(121-160 → 100-180)로 작은 번호 자주 통과.
- S33 풀 외 차단으로 학설 풀 외 번호 절대 안 나옴 → 분포 부자연.

#### 새 architecture (S43 단일 추첨)

1. **단일 weight 벡터**: 모든 활성 strategy의 가중을 1-45 base 1.0 + 학설/통계/Luck 보너스 +0.3~+0.5로 합성.
2. **`weightedSample` 1번 호출**: 6번호 단일 추첨. 인접 클러스터링 자연 해소.
3. **다중 분배 폐기**: `distributeCounts` / `computeStrategyContext` / 잘라쓰기 휴리스틱 모두 우회.
4. **풀 컷팅 폐기**: 학설 = 정의가 아닌 약한 가중치. 1-45 전체 추첨 가능.
5. **Luck 보너스 약화**: 시드 6번호 +0.5 (Luck 비례). base 1.0의 1.5배.
6. **balancer post-filter 폐기**: 합 필터 안 함. 자연 통과 빈도.
7. **풀 외 차단(S33) 폐기**: 학설 가중일 뿐. 풀 외도 base weight로 추첨.

#### 5.2.5.1. 모델

- **조립식 = strategy 조합 + 캐릭터 시드 + 회차**.
- 미리보기 단계 없음. 사용자가 strategy 토글로 조립식 정의 → **"+ 1세트"** 또는 **"+ 5세트"** 클릭 = 즉시 추천1, 추천2, ...로 누적 push.

- 같은 numbers 조합은 자동 skip (중복 차단).
- 결정론: 같은 strategy 조합 + 같은 시점 클릭 = 같은 결과. 미리보기 없어도 strategy 토글 상태가 결과를 1대1 결정.

### 5.3.1. 통계 페이지 갱신 흐름

5.3.1.1. **메타 라인**: 헤더 아래에 "{최신 회차}회까지 반영 · 최근 추첨 YYYY-MM-DD" 표시. 사용자가 한눈에 데이터 신선도 판단.

5.3.1.3. **명시 갱신 버튼**: 헤더 우측 "↻ 갱신". 클릭 시 같은 흐름이지만 결과를 항상 토스트로 통보.

### 5.4. 결정론성

- **시드 의존 전략(5개, S34 짝꿍 폐기로 6 → 5)**: 같은 캐릭터 + 같은 회차 + 같은 토글 = 같은 결과. blessed / astrologer / intuitive / zodiacElement / fiveElements.

### 7.5. 백캐스트 (Luck 부트스트랩)

추천 회차는 항상 **다음 추첨(미래)**이고 발표 전이라 매칭이 영원히 안 일어난다 (4.x). 이를 보강하기 위해 캐릭터 첫 추첨 탭 진입 시 최근 N회(`BACKFILL_RECENT_COUNT`, docs/02_data.md 1.16)에 대해 결정론적 추천을 history에 backfill + 매칭한다.

7.5.1. 위치: `core/history.js` `backfillRecommendations(character, draws, strategyId, stats, lastN)`.
7.5.2. 결정론: 같은 캐릭터 + 같은 draws + 같은 통계 = 같은 백필 결과 (같은 시점 통계를 모든 회차에 사용).
7.5.3. idempotent: 이미 history에 있는 회차는 건너뜀. 매 진입마다 호출해도 부담 없음.
7.5.4. 매칭은 즉시. 적중된 회차는 `applyLuckGrowth`가 다음 단계에서 luck 보너스 적용.
7.5.5. 사용자가 다음 추첨 회차를 1회 보면 backfill 30회 + 다음 회차 1개 = history 31개.
7.5.6. 통계 정확도 vs 단순성 트레이드: 회차별 그 시점의 통계가 아니라 현재 시점 통계를 모든 백필에 사용. 의미는 "이 캐릭터의 현재 능력으로 과거 N회를 시뮬한다면". 회차별 시점 통계 백캐스트가 필요해지면 별도 모듈로 분리.
