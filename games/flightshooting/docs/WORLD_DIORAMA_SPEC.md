## 1. 렌더링 기준

### [실측]
- 반응형 해상도. Canvas 내부 = `floor(W*dpr) × floor(H*dpr)`, `dpr = min(devicePixelRatio, 2)`. 논리 좌표 = CSS 픽셀(`ctx.setTransform(dpr,…)`). (`main.js resize()`)
- 루프는 requestAnimationFrame 기반(`shared/loop.js`).

### [제안]
- Design Resolution: 논리 720 × 1280(세로 9:16) 기준으로 에셋·좌표 설계.
- Target FPS 60 / Minimum 30(이하 시 dt 보정으로 스크롤 속도 유지). DPR 정책은 실측대로 2 캡 유지.

### 제안 근거
- 반응형이라 물리적 고정 해상도가 없어, 에셋·좌표를 위한 "논리 기준"이 필요하다. 9:16은 현재 세로 플레이와 dpr 2 캡에 맞는 흔한 모바일 세로비.

### 아직 확인할 사항
- 실제 배포 대상 기기의 세로비 분포(레터박스 허용 범위). 데스크톱 최대 세로 높이 상한.

## 2. 디오라마 원본 규격 (방식 비교)

### [실측]
- 세로 청크를 이어 붙이는 구조는 현재 없다. `drawScrollingBg`는 단일 이미지를 세로로 무한 타일 반복만 한다. 어느 방식이든 렌더러 신규 개발이 필요하다.

### [제안]
- 주 방식 **B(세로 청크 연결)** + 보조 **C(랜드마크·효과 레이어)**.
- 예상 구조(검증 대상): 국가당 세로 청크 3~6개(지형+일반건물+기본조명 사전 합성, 불투명) + 인트로 대상 랜드마크만 알파 레이어 + 공용 안개·구름그림자 효과 레이어.

### 제안 근거
- 청크(B)가 모바일 메모리·부분 수정·로딩에 가장 안전하고, 인트로 줌은 랜드마크만 클로즈업이라 그 랜드마크를 레이어(C)로 분리해야 줌 대상이 잡힌다. 아트 기준의 "런타임 레이어를 무조건 5개로 두지 말고 사전 합성"과도 일치한다(제작 단계 Terrain→…→Lighting ≠ 런타임 레이어 수).

### 아직 확인할 사항
- 청크 경계 연결 방식(단순 이어붙이기 vs 오버랩 블렌드) — 이음매 가시성은 프로토타입 실측.
- 랜드마크를 별도 레이어로 둘 범위(인트로 대상 1개만 vs 다수).

## 3. 인트로 줌아웃 수치

### [실측]
- 인트로 동안 스폰·진행이 멈추고 나머지는 진행된다(`introTimer`, `world.js stepWorld`). 현 `stageIntro = 2.2s`(`numbers.js:240`). `drawImage`/`ctx.scale`로 확대·축소 렌더 가능.

### [제안] (전부 provisional)
- 인트로 시작 배율 2.4x → 플레이 배율 1.0x, 줌아웃 1.8s(2.2s 인트로 내 완료), easing easeOutCubic.
- 줌 중심 = 랜드마크 좌표. 랜드마크 화면 점유 인트로 60~70% → 플레이 하단 원경 15~25%.
- 플레이어 등장·적 스폰·입력·스크롤 = 줌아웃 완료 시점. HUD·국가 배너 = 인트로부터.

### 제안 근거
- 인트로 정지 로직이 이미 있어 카메라 스케일 보간만 얹으면 된다. 방향은 아트 기준(인트로=가까움, 플레이=멀어짐, 컷 전환 금지)에 맞춘다. `ctx.scale`은 "거리(스케일)"만 바꾸므로, 비스듬한 "각도"는 이미지 자체가 프리렌더로 담아야 한다(2D 스케일로는 각도가 안 바뀜).

### 아직 확인할 사항
- 배율·시간·easing 체감값(아트 가독성·랜드마크 인식 시간 검증 후 확정).
- 줌 중심을 화면 어디에 둘지(랜드마크가 좌우 가장자리 배치라는 아트 기준과 줌 중심의 관계).

## 4. 스테이지 길이와 스크롤 속도

### [실측]
- 스테이지는 시간 고정이 아니다. 웨이브가 순차 스폰되고(마지막 웨이브 등장 ≈ 16~21초, `waves.js buildWaves`), 잔적 처치 후 보스 pending 1.4s → 보스전으로 끝난다(`world.js checkProgress`). 배경은 이와 독립적인 시간 기반 무한 스크롤. 현 배경 스크롤 `BG_SCROLL = 22px/s`(원경 대기용, 느림).

### [제안]
- 디오라마 전환 시 스테이지 길이 = "배경 세로 통과 길이"로 재정의. `구간 픽셀길이 = 구간시간 × scrollSpeed`, 총 세로 = Σ 구간. 한국 5구간(전통→도심→한강→고층→항구)을 진행률에 매핑. `scrollSpeed`는 구간별 조정 가능하게 데이터화(§7).

### 제안 근거
- 현재 "웨이브 소진+보스"로 끝나는 진행을 배경 통과 거리와 묶어야 "도심 진입 시 도심 적 등장" 같은 연출이 가능하다. 웨이브 t(등장 시각)를 구간 경계에 맞추면 기존 스폰 로직을 그대로 재사용한다.

### 아직 확인할 사항
- 구간별 목표 시간·스크롤 속도(플레이 시간·아트 밀도 검증 후). 보스전을 배경 통과 후 어디에 둘지.

## 5. 이미지 제작 규격

### [실측]
- 포맷·크기 제약 코드에 없음. canvas `drawImage`는 straight alpha 가정.

### [제안]
- 포맷 WebP 우선(PNG 폴백, AVIF 후순위). 폭 1080px(논리 540 × dpr 2). 청크 높이 2048px 기본·최대 4096px. 청크당 ≤ 400KB(WebP q78~82). sRGB. 배경 청크 불투명(알파 없음), 랜드마크·효과만 알파. straight alpha로 export. 모바일용 0.5x 다운스케일 버전 제공. 파일명 `diorama/<countryId>/chunk_NN|landmark_*|effect_*`, 공용 `diorama/_common/`.

### 제안 근거
- WebP는 용량·알파를 만족하고 호환도 넓다. 청크 높이 상한은 모바일 텍스처·메모리 안전 범위. 배경 불투명은 용량·합성 비용 절감. 브라우저 자동 밉맵이 없어 다운스케일 버전이 필요하다.

### 아직 확인할 사항
- 실제 청크 높이·용량(모바일 디코드·메모리 프로파일 후). 다운스케일 버전을 어느 화면폭·dpr에서 전환할지.

## 6. 좌표 및 안전 영역

### [실측]
- 플레이어 초기 세로 `yRatio 0.82`, 보스 등장 `spawnTop 62`(`numbers.js`). HUD는 상단 밴드.

### [제안] (논리 720×1280 기준)
- 플레이 가능: 전체 폭, 세로 상단 8%~하단 6% 제외. 랜드마크 배치: 하단 원경(하위 25%) 또는 좌우 가장자리. 중앙 탄막 가독 보호: 세로 중앙 60%(저대비·디테일 억제). UI 가림: 상단 7% HUD 밴드. 적 스폰 금지: HUD 밴드+플레이어 근접. 보스 연출: 상단 중앙. 인트로 포커스 좌표 / 줌 종료 기준 좌표는 국가 데이터로.

### 제안 근거
- 중앙 60% 보호는 docs 배경 규칙과 아트 가독성 기준(중앙 전투 영역 보호)의 공통값. 랜드마크를 좌우·전환 지점에 두라는 아트 기준을 좌표로 옮긴 것.

### 아직 확인할 사항
- 디오라마 원근 때문에 "하단=가까움"이 되는데, 랜드마크를 하단 원경에 둘지 좌우에 둘지(아트 배치 기준과 조율).

## 7. 국가 데이터 구조

### [실측]
- 현재 배경은 데이터 기반이 아니라 코드 하드코딩(SCENES 서울)+공용 우주. 국가 배경 메타 구조 없음.

### [제안]
```json
{
  "countryId": "KR", "displayName": "대한민국", "stageId": "KR_01",
  "introFocus": { "x": 360, "y": 980, "zoom": 2.4 },
  "gameplay": { "zoom": 1.0, "scrollSpeed": 60, "stageLength": 2900 },
  "chunks": [ { "src": "diorama/KR/chunk_00.webp", "h": 2048 } ],
  "landmarks": [ { "src": "diorama/KR/landmark_gyeongbokgung.webp", "x": 300, "y": 1050, "scale": 1.0, "layer": "mid" } ],
  "effects": [ { "src": "diorama/_common/cloud_shadow.webp", "mode": "scroll", "speed": 12, "alpha": 0.35 } ]
}
```
- `chunks`는 아래→위로 이어 붙여 스크롤(B), `landmarks`·`effects`는 레이어 합성(C). 좌표는 논리 720×1280.

### 제안 근거
- `scrollSpeed`·`stageLength`를 데이터화해 §4 계산 구조와 국가별 조정을 코드 수정 없이 가능하게. 기존 `countries.js`(라우팅)와는 분리(배경 전용 메타).

### 아직 확인할 사항
- `countries.js`/`countries.education.js`와의 연결 키(countryId ↔ ko). 랜드마크 좌표계 원점 규약.

## 8. 대한민국 최소 프로토타입

### [실측]
- 기존 전투 로직(발사·웨이브·충돌)은 배경과 독립이라 그대로 재사용 가능. 인트로 정지·drawImage 스케일 모두 확인됨.

### [제안]
- 목적 3가지 검증만: (1) 인트로 클로즈업→플레이 줌 연결 자연스러움 (2) 세로 청크 연결 이음매 비가시성 (3) 디오라마 위 탄환·적·플레이어 가독성.
- 범위: 한국, 인트로 랜드마크 경복궁 1개, 플레이 약 3화면 길이, 청크 2~3개, 보스·교육 UI·투어·최종 폴리싱 없음, PC·모바일 성능 비교, 전투 로직 그대로.

### 제안 근거
- "예쁜 완성본"이 아니라 렌더 구조·가독성 검증용 최소 샘플이어야 위험을 싸게 조기 발견한다(아트 기준의 프로토타입 방향과 일치).

### 아직 확인할 사항
- 프로토타입 청크 2개 vs 3개(연결 검증엔 최소 2개 경계 필요). 임시 에셋(placeholder)으로 먼저 구조 검증 후 실제 아트 투입 여부.

## 9. 미결정 항목 (프로토타입 전 결정 필요)

- 논리 기준 해상도 확정(720×1280 채택 여부) — 좌표·에셋 전제.
- 청크 폭·높이 1세트 확정 — 프로토타입 제작 착수 전제.
- 줌 배율·시간 초깃값 — 연출 프로토타입 전제.
- 스크롤 속도 초깃값 — 청크 개수·길이 계산 전제.
- 청크 경계 연결 방식(이어붙이기 vs 오버랩).
- 카메라 프리렌더 각도(아트 결정값이나 좌표·랜드마크 배치에 직접 영향).
- 배경 전용 메타를 `countries.js`에 합칠지 별도 파일로 둘지.

## 10. 리스크와 검증 방법

- **원근 고정 한계(핵심)**: 2D 프리렌더는 스크롤 중 원근이 바뀌지 않는다(진짜 3D 카메라 아님). 청크가 위로 흘러도 각 청크의 각도·원근은 고정이라 "카메라가 전진하는" 입체감이 약할 수 있다. → 검증: 프로토타입에서 청크 3개 스크롤 실제 체감.
- **청크 이음매**: 경계에서 지형·조명·도로 연결이 어긋나면 눈에 띈다. → 검증: 이음매 3회 이상 통과 캡처, 오버랩 블렌드 비교.
- **모바일 메모리·디코드**: 큰 청크·다운스케일 미비 시 모바일 스터터. → 검증: 실기기 FPS·메모리 프로파일.
- **가독성**: 디오라마 위 탄환·적 대비 저하. → 검증: 전투 밀집 상황 캡처(중앙 60% 보호 실효 확인).
- **줌 중 해상도 저하**: 인트로 2.4x 확대 시 픽셀 뭉개짐. → 검증: 랜드마크 레이어를 확대 배율 고려한 고해상도로 제작하거나 인트로 전용 고해상 컷 사용.
- **50개국 로딩**: 전량 프리로드 시 메모리 초과. → 검증: 현재+인접 스테이지만 로드/언로드(LRU) 전략 프로토타입.

## 11. 프로토타입 기준값 (Prototype Baseline v0.1)

전부 `[프로토타입 기준]` — 기술 테스트 전용 승인 임시값. 프로토타입 검수 후 FINAL/REVISE/REJECT로 재분류한다. 지금은 최종 확정값 없음.

- 해상도: 논리 720×1280, dpr 2캡, 60fps(min 30), 9:16, 반응형 표시. 게임·배경 메타 좌표 모두 논리 720×1280 기준.
- 청크: 폭 1080 × 높이 2048, 불투명 WebP q80, 한국 2~3개, 동일 시점·조명·원근·건물 스케일.
- 청크 연결: 오버랩 96~160px 크로스페이드. 블렌드 구간엔 랜드마크 금지, 도로·강·평지·숲 등 연결 쉬운 환경만. 블렌드 비용이 크면 오버랩 alpha를 청크에 사전 굽는다(런타임 마스크 회피).
- 카메라: 비스듬한 탑다운, roll 0. 인트로·플레이 각도 동일, 거리/crop scale만 변경. 원근 투영 전 청크 동일. **각도는 숫자로 정하지 않고 최신 승인 기준 이미지의 인상을 따른다**(아트 SSOT `ChatGPT_World_Diorama_Art_MD_MASTER/STYLE_GUIDE.md` 카메라 절). pitch 숫자·시안 비교는 폐기.
- 인트로 줌: 시작 2.4x → 플레이 1.0x, 1.8s, easeOutCubic. 인트로 중 진행 정지, 줌 완료 후 스크롤·적 스폰·입력 시작. 플레이어는 줌 종료 0.2~0.3s 전 등장 가능. HUD는 줌 종료 직전/시. 인트로 전용 별도 배경 교체 금지(같은 좌표 확대→1.0x).
- 포커스: 줌 중심 = 랜드마크 월드 좌표. 줌아웃하며 카메라 중심을 gameplay 기준 중심으로 보간(단순 ctx.scale 금지, 포커스+중심 이동 병행).
- 랜드마크: 동일 렌더 소스의 고해상 투명 레이어 허용(2.4x 뭉개짐 방지). 인트로 전용 다른 그림·다른 각도 금지, 배경 속 위치와 정합.
- 스크롤 속도: 기존 게임엔 배경 스크롤이 사실상 없음([실측] 별 40~166px/s, 배경 BG_SCROLL 22 — A안에서 추가). baseline은 프로토타입에서 전투 템포 유지 기준으로 잡고 slow 0.85 / normal 1.0 / fast 1.15 프리셋을 개발자 옵션으로 즉시 비교, 검수 후 택1. 배경을 위해 전투 속도 변경 금지.
